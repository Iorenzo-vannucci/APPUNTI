     Signals: in parole povere….

• forma di IPC limitata, nel senso per esempio che
  non contengono ulteriore informazione (il dato
  scambiato è il signal stesso), Real-Time a parte
  (vedi oltre)

• sono simili ad "interrupt software" o, se preferite,
  a notifiche, che vengono inviati ad un processo
  (od a un thread) al verificarsi di un evento

• gli standard hanno un significato intrinseco e,
  diciamo così, "tradizionale"...
                                                     4
           Signals: cosa accade

• il flusso di esecuzione del processo viene
  interrotto dal sistema operativo per eseguire la
  gestione del signal:
  – gestione definita dal processo (se lo ha fatto)
  – gestione di default (altrimenti)


• la gestione può essere varia: il processo termina,
  viene sospeso, viene eseguita una funzione
  registrata dal processo all'uopo, il signal viene
  ignorato, etc.
                                                       5
  Signals: gestioni di default (da man 7 signal)

Signal dispositions
   • Se non diversamente specificato ogni signal ha una
       disposition di default. Le disposition di default
       possono essere:
     o   Term      Default action is to terminate the
                   process.
     o   Ign       Default action is to ignore the signal.
     o   Core      Default action is to terminate the process and
                   dump core (see core(5)).
     o   Stop      Default action is to stop the process.
     o   Cont      Default action is to continue the process if it is
                   currently stopped.

             Signals: sono inviati da…
The following system calls and library functions allow the caller to send a
signal:
raise(3)          Sends a signal to the calling thread.
kill(2)           Sends a signal to a specified process, to all members of
                  a specified process group, or to all processes on the
                  system.
pidfd_send_signal(2) Sends a signal to a process identified by a PID file
                  descriptor.
killpg(3)         Sends a signal to all of the members of a specified
                  process group.
pthread_kill(3) Sends a signal to a specified POSIX thread in the same
                  process as the caller.
tgkill(2)         Sends a signal to a specified thread within a specific
                  process. (This is the system call used to implement
                  pthread_kill(3).)
sigqueue(3)       Sends a real-time signal with accompanying data to
                  a specified process.

             Signals: chi li riceve
• process-directed o thread-directed
  – process-directed:
     • inviati all'intero processo
     • per esempio generati dal kernel per motivi diversi da
       un'eccezione HW; o perché inviato con kill() o sigqueue()
     • ricevuti da uno dei thread che non li bloccano. Se più di uno
       è il kernel che sceglie a quale thread destinarli
  – thread-directed:
     • inviati al thread
     • per esempio perché generati da una specifica istruzione del
       linguaggio macchina che ha provocato un'eccezione HW (per
       es.: segmentation fault, etc.); o inviati con pthread_kill(),
       tgkill()

             Signals: signal mask
• un signal può essere bloccato, cioè non sarà ricevuto
  fino a che non verrà sbloccato

• la signal mask definisce quali signal sono bloccati e
  quali no

• una signal mask per ogni thread (per-thread)

• SIGKILL e SIGSTOP non possono essere gestiti, ignorati
  o bloccati

• ereditata dai child dopo fork(2); preservata dopo
  execve(2)
                                                      14
        Signals: segnali pendenti
• nel tempo tra la generazione e l'invio al processo
  il signal è detto pendente

• può essere pendente perché bloccato

• un thread può ottenere la lista dei segnali
  pendenti con sigpending()

• la lista dei segnali pendenti è resettata (vuota)
  dopo fork(2) ed è preservata dopo execve(2)

       sigset_t sono disponibili delle funzioni apposite. Vedi man 3 sigsetops
       e slide successive

oldset: If oldset is non-NULL, the previous value of the signal mask is stored in
      oldset

    Signals: cambiare la gestione (disposition) corrente
•    per ogni signal ogni processo può cambiare la gestione corrente

•    si fa con sigaction(2) e signal(2) (vedi oltre per il dettaglio di sigaction(2)). La
     seconda è meno portabile, meglio usare l'altra

•    tre opzioni:
           – ignorare il signal
           – lasciare/ripristinare la gestione di default del kernel
           – specificare una funzione handler che gestisca il segnale

•     un handler è una funzione definita dal programmatore che il sistema invoca alla
      ricezione del signal (callback) in maniera asincrona (il processo/programmatore
      non sa quando sarà invocata)

•     la disposition (gestione) di un signal è un attributo del processo condiviso da tutti
      i suoi threads

•     le disposition sono duplicate da fork(2) nel child. Dopo execve(2) le disposition
      che prevedono un handler sono resettate al default, quelle che prevedono di
      ignorare rimangono inalterate
                                                                                            21
      Cambiare la disposition: sigaction(2)
#include <signal.h>
int sigaction(int signum,
              const struct sigaction *act,
              struct sigaction *oldact);
Usata per variare la gestione di un signal da parte di un processo

• signum: specifica il signal di cui variare la gestione
• act: struct che definisce la nuova gestione
• oldact: struct che riceve le informazioni relative alla gestione
  precedente

Ritorna 0 (successo) o -1 (errore; errno settata)

     Registrare Signal Handlers: sigaction(2)
struct sigaction {
   void     (*sa_handler)(int);
   void     (*sa_sigaction)(int, siginfo_t *, void *);
   sigset_t   sa_mask;
   int        sa_flags;
   void     (*sa_restorer)(void);
};
sa_handler: indica la nuova gestione
      – SIG_IGN: ignorato
      – SIG_DFL: funzione di default
      – puntatore a funzione dell'handler. Riceve come unico argomento il numero del
        signal ricevuto

SIGKILL e SIGSTOP non possono essere gestiti, ignorati o bloccati

N.B.: non assegnare contemporaneamente sa_handler e sa_sigaction (in certe
architetture sono oggetto di union). In questa sede non parliamo di sa_sigaction
Per maggiori informazioni vedere man sigaction(2)

Signals: passi per l'esecuzione dell'handler
  Steps (sempificati...):
  • il kernel esegue operazioni                          preparatorie           per
     l'esecuzione dell'handler:
      –    rimuove il signal dalla lista dei pendenti
      –    salva lo stato del processo/thread da interrompere (i
           registri, etc.)
  •   l'handler viene eseguito in user-mode
  •   quando l'handler ritorna lo stato del processo/thread
      viene ripristinato
  •   il processo/thread che ha ricevuto il signal continua
      la sua esecuzione

  Nota: dal punto di vista del kernel non c'è differenza tra eseguire l'handler o il
  codice del processo/threads. Nessuna ulteriore "precauzione" viene presa dal
  kernel e le informazioni sullo stato del processo sono salvate nei registri e nello
  stack del processo (non nelle aree di memoria del kernel). Per maggiori dettagli
  vedi man 7 signal alla sezione "Execution of signal handlers "

 Signal Handlers. Esempio1: sigaction()
#include<stdio.h>
#include<string.h>
#include<stdlib.h>
#include<signal.h>
#include<unistd.h>

void SigIntHandler(int iSignalCode) {
  printf("Handler: ricevuto signal %d (%s)\n", iSignalCode, strsignal(iSignalCode));
  return;
}
int main() {
    struct sigaction sa;
    int i;
    memset(&sa, '\0', sizeof(struct sigaction));
    sa.sa_handler = SigIntHandler;    // Provate a variare questa assegnazione
                                      // come specificato nelle due righe seguenti
    //sa.sa_handler = SIG_DFL;
    //sa.sa_handler = SIG_IGN;
    sigaction(SIGINT, &sa, NULL);
    printf("Premere Ctrl+C per eseguire la \"disposition\" di SIGINT\n");
    printf("Se avete definito un handler per Ctrl+C o l'avete ignorato,\n");
    printf("Potete uscire con Ctrl+\\ (provoca l'invio di SIGQUIT)\n");
    for(i = 1; ; i++) {
           sleep(1);
           printf("%d\n", i);
    }
    return EXIT_SUCCESS;
}

                                      Signals: Esempio 3
•   SIGALRM gestito, SIGINT mascherato. Attesa sincrona specifica (sigwait())
#include<stdio.h>
#include<unistd.h>
#include<string.h>
#include<signal.h>

void SigHandler(int iSignalCode) {
  printf("Handler: ricevuto signal %d: %s\n", iSignalCode, strsignal(iSignalCode));
  return;
}

int main (void) {
            struct sigaction sa;
            int i, signum;
            sigset_t sigset;

            // Gestione con handler di SIGALRM
            memset(&sa, '\0', sizeof(struct sigaction));
            sa.sa_handler = SigHandler;
            sigaction(SIGALRM, &sa, NULL);

            // Mascheramento di SIGINT
            sigemptyset(&sigset);
            sigaddset(&sigset, SIGINT);
            sigprocmask(SIG_BLOCK, &sigset, NULL);

            alarm(3);
            for(i = 1; i <= 4; i++) {
                        printf("Main Sigwait n. %d\n", i);
                        sigwait(&sigset, &signum);
                        printf("Main Sigwait: ricevuto signal %d; %s\n", signum, strsignal(signum));
            }
            printf("Ciao!\n");
            return 0 ;
}

  accodate

• invece, per quelli standard, se più istanze dello

         Real-Time (RT) Signals
• se inviati con sigqueue(3) possiamo
  "abbinare" al signal un valore (int o pointer)
  che sarà inviato insieme al signal

• il processo destinatario può allora registrare
  un handler col flag SA_SIGINFO in sigaction() e
  può ottenere questo dato nel campo si_value
  della struttura siginfo_t

                                                   46
         Real-Time (RT) Signals
• l'ordine in cui i RT Signal sono ricevuti è lo
  stesso in cui sono stati inviati
• più segnali RT dello stesso tipo sono ricevuti
  nell'ordine in cui sono stati inviati
• se segnali RT di tipo diverso sono inviati ad un
  processo sono ricevuti a partire dal numero di
  segnale più basso
• per quelli standard l'ordine di ricezione dei
  segnali pending non è specificato

  printf("Handler: intero passato %d\n", si->si_value.sival_int);
  return;
}
int main (void) {
               struct sigaction sa;
               int i, signum;
               sigset_t sigset;
               union sigval value;

               // Gestione con handler di SIGRTMIN + 1
               memset(&sa, '\0', sizeof(struct sigaction));
               sa.sa_sigaction = SigHandler;
               sa.sa_flags = SA_SIGINFO;
               sigaction(SIGRTMIN + 1, &sa, NULL);

               // Mascheramento di SIGINT
               sigemptyset(&sigset);
               sigaddset(&sigset, SIGINT);
               sigprocmask(SIG_BLOCK, &sigset, NULL);

               sleep(3);
               value.sival_int = 999;
               sigqueue(getpid(), SIGRTMIN + 1, value);
               for(i = 1; i <= 3; i++) {
                              printf("Sigwait n. %d\n", i);
                              sigwait(&sigset, &signum);
                              printf("Sigwait: ricevuto signal %d: %s\n", signum, strsignal(signum));
               }
               printf("Ciao!\n");
               return 0 ;
}
